# Action & Input 컴포넌트

## 버튼 (Button)

### 용도

사용자가 동작을 실행하거나 흐름을 진행하기 위해 클릭하는 대화형 요소. Primary, Secondary, Tertiary, Text 4단계 계층이 있다.

### HTML 스니펫

```
html/code/button.html
html/code/button_size.html
html/code/button_hierarchy.html
html/code/button_icon.html
html/code/button_text.html
html/code/button_with_icon.html
```

### 구조

- 버튼 요소 (`krds-btn`)
  - 아이콘 (선택)
  - 레이블 텍스트

### 유형

| 유형 | CSS 클래스 | 용도 |
|------|-----------|------|
| Primary | `krds-btn-primary` | 화면의 가장 중요한 동작 (1개만) |
| Secondary | `krds-btn-secondary` | 보조 동작 |
| Tertiary | `krds-btn-tertiary` | 낮은 강조 동작 |
| Text | `krds-btn-text` | 최소 강조 동작 |

### 사용성 가이드라인

1. 버튼 그룹 내 모든 버튼은 동일한 크기를 사용한다.
2. 한 화면에서 최상위 강조(Primary) 버튼은 1개만 사용한다.
3. 수직으로 버튼을 배치할 때, 가장 강조된 버튼을 첫 번째(최상단)에 배치한다.
4. 버튼 레이블은 동사형으로 작성한다 (예: "저장", "삭제", "제출하기"). 모호한 레이블("확인", "예")을 피한다.
5. 더블클릭 방지 처리를 적용한다. 폼 제출 버튼 등에서 중복 실행을 방지한다.
6. **탐색 상호작용**: 포커스/호버 시 시각적 피드백(색상 변화)을 제공한다.
7. **실행 상호작용**: 클릭 또는 Enter/Space 키로 동작을 실행한다.

### 접근성 가이드라인

- `<button>` 태그를 사용한다. `<a>` 태그나 `<div>`로 버튼을 만들지 않는다.
- 아이콘만 있는 버튼에는 `aria-label`을 제공한다.
- 비활성 버튼에 `disabled` 속성을 사용한다 (`aria-disabled`가 아닌).
- 로딩 중인 버튼에 `aria-busy="true"`를 적용한다.
- KWCAG 2.2 / WCAG 2.1 준수: 4.1.2 Name, Role, Value, 2.1.1 Keyboard

### 상호작용

- **클릭/Enter/Space**: 동작 실행
- **Tab**: 다음 대화형 요소로 포커스 이동
- **호버**: 배경색 변화 (Hover 상태 색상)
- **Pressed**: 배경색 더 진해짐

### 플랫폼 고려사항

- 모바일에서 터치 영역을 최소 44x44px로 확보한다.
- 모바일에서 전체 너비(full-width) 버튼 사용을 권장한다.

## 링크 (Link)

### 용도

다른 페이지나 리소스로 이동하기 위한 텍스트 요소. 동작 실행이 아닌 탐색 목적으로 사용한다.

### HTML 스니펫

```
html/code/link.html
```

### 구조

- 링크 요소 (`<a>`)
  - 텍스트
  - 외부 링크 아이콘 (선택, `ico_go.svg`)

### 사용성 가이드라인

1. 인라인 링크는 주변 텍스트와 동일한 크기를 사용한다. 링크만 크거나 작게 하지 않는다.
2. 색상만으로 링크를 구분하지 않는다. 밑줄 또는 아이콘 등 추가적인 시각적 구분을 제공한다.
3. 링크 텍스트는 정확한 목적지를 설명한다. "여기", "클릭", "더보기" 같은 모호한 텍스트를 피한다.
4. 이메일 주소 링크는 `mailto:` 프로토콜을 사용한다.
5. 전화번호 링크는 `tel:` 프로토콜을 사용한다.
6. 외부 사이트 링크는 내부 링크와 시각적으로 구분한다 (외부 링크 아이콘 `ico_go.svg` 등).
7. 외부 링크를 구분하는 방식은 사이트 전체에서 일관되게 사용한다.
8. 모든 링크가 정확한 목적지로 연결되는지 확인한다. 깨진 링크가 없어야 한다.
9. **탐색 상호작용**: 포커스/호버 시 밑줄 또는 색상 변화로 피드백을 제공한다.
10. **실행 상호작용**: 클릭 또는 Enter 키로 이동한다.

### 접근성 가이드라인

- `<a href="...">` 태그를 사용한다. `href` 없는 `<a>` 태그를 사용하지 않는다.
- 새 탭으로 열리는 링크에 `target="_blank"`와 함께 "(새 창)" 텍스트를 `aria-label`에 포함한다.
- 외부 링크 아이콘에 `aria-hidden="true"`를 적용하고, 접근 가능한 이름에 외부 링크임을 명시한다.
- KWCAG 2.2 / WCAG 2.1 준수: 2.4.4 Link Purpose, 3.2.5 Change on Request

### 상호작용

- **클릭/Enter**: 목적지로 이동
- **Tab**: 다음 링크/대화형 요소로 포커스 이동
- **호버**: 밑줄 표시 또는 색상 변화

### 플랫폼 고려사항

- 모바일에서 전화번호 링크(`tel:`)가 올바르게 동작하는지 확인한다.

## 텍스트 입력 (Text Input)

### 용도

사용자가 짧은 텍스트를 입력하는 단일 행 입력 필드. 이름, 이메일, 검색어 등에 사용한다.

### HTML 스니펫

```
html/code/text_input.html
html/code/text_input_icon.html
html/code/text_input_size.html
html/code/text_input_state.html
```

### 구조

- 입력 필드 그룹 (`krds-input`)
  - 레이블 (`<label>`)
  - 도움말 텍스트 (선택)
  - 입력 필드 (`<input type="text">`)
  - 아이콘 (선택, 검색 등)
  - 오류 메시지 (선택)

### 사용성 가이드라인

1. 입력 필드는 컨테이너 전체 너비를 사용한다. 예상 입력 길이에 맞게 너비를 조절한다.
2. 모든 입력 필드에 시각적 레이블(`<label>`)을 제공한다. 레이블을 생략하지 않는다.
3. 플레이스홀더 텍스트로 레이블을 대체하지 않는다. 플레이스홀더는 보조 힌트로만 사용한다.
4. 복사/붙여넣기를 허용한다. 입력 필드에서 붙여넣기를 막지 않는다.
5. 적절한 경우 자동완성(`autocomplete`) 속성을 제공한다 (이름, 이메일, 주소 등).
6. **데이터 입력 상호작용**: 포커스 시 입력 가능 상태로 전환, 입력 내용 실시간 표시.

### 접근성 가이드라인

- `<label for="id">`로 레이블과 입력 필드를 연결한다.
- 필수 입력 필드에 `required` 속성과 시각적 표시(*)를 제공한다.
- 오류 메시지를 `aria-describedby`로 입력 필드와 연결한다.
- 오류 상태의 입력 필드에 `aria-invalid="true"`를 적용한다.
- 적절한 `type` 속성을 사용한다 (`email`, `tel`, `url` 등).
- KWCAG 2.2 / WCAG 2.1 준수: 1.3.1 Info and Relationships, 3.3.2 Labels or Instructions, 3.3.1 Error Identification

### 상호작용

- **클릭/Tab**: 입력 필드에 포커스
- **입력**: 텍스트 입력 및 실시간 표시
- **Enter**: 폼 제출 (폼 내부인 경우)
- **Esc**: 입력 취소 또는 자동완성 닫기

### 플랫폼 고려사항

- 모바일에서 적절한 키보드 유형이 표시되도록 `inputmode` 속성을 사용한다 (`numeric`, `email`, `tel` 등).

## 텍스트 영역 (Textarea)

### 용도

사용자가 여러 줄의 텍스트를 입력하는 다중 행 입력 필드. 의견, 설명, 메모 등 긴 텍스트에 사용한다.

### HTML 스니펫

```
html/code/textarea.html
```

### 구조

- 텍스트 영역 그룹
  - 레이블 (`<label>`)
  - 도움말 텍스트 (선택)
  - 텍스트 영역 (`<textarea>`)
  - 글자수 카운터 (선택)
  - 오류 메시지 (선택)

### 사용성 가이드라인

1. 모든 텍스트 영역에 시각적 레이블(`<label>`)을 제공한다.
2. 플레이스홀더 텍스트로 레이블을 대체하지 않는다.
3. 복사/붙여넣기를 허용한다.
4. 최대 글자수 제한이 있는 경우 글자수 카운터를 표시한다 (예: "0/500").
5. **데이터 입력 상호작용**: 포커스 시 입력 가능 상태로 전환, 여러 줄 입력 가능.
6. **스크롤 상호작용**: 내용이 영역을 초과하면 내부 스크롤을 제공한다.
7. 텍스트 영역의 전체 크기(높이)를 적절히 설정한다. 최소 3~4행을 표시할 수 있는 높이를 기본으로 한다.

### 접근성 가이드라인

- `<label for="id">`로 레이블과 텍스트 영역을 연결한다.
- 필수 입력에 `required` 속성을 적용한다.
- 글자수 카운터 변경 시 `aria-live="polite"` 영역에서 스크린 리더에 알린다.
- 오류 메시지를 `aria-describedby`로 연결한다.
- KWCAG 2.2 / WCAG 2.1 준수: 1.3.1 Info and Relationships, 3.3.2 Labels or Instructions

### 상호작용

- **클릭/Tab**: 텍스트 영역에 포커스
- **입력**: 텍스트 입력, 줄바꿈(Enter)
- **스크롤**: 내용 초과 시 내부 스크롤

### 플랫폼 고려사항

- 모바일에서 텍스트 영역 크기를 충분히 확보한다.
- `resize` CSS 속성으로 사용자가 영역 크기를 조절할 수 있도록 허용한다 (수직 방향만 권장).

## 날짜 입력 (Date Input)

### 용도

사용자가 특정 날짜를 입력하는 컴포넌트. 년, 월, 일을 개별 필드 또는 달력 피커로 입력한다.

### HTML 스니펫

```
html/code/date_input.html
```

### 구조

- 날짜 입력 그룹
  - 레이블 (`<label>`)
  - 도움말 텍스트 (형식 안내)
  - 년/월/일 입력 필드 (개별 `<input>`)
  - 달력 아이콘 버튼 (선택, `ico_calendar.svg`)
  - 오류 메시지 (선택)

### 사용성 가이드라인

1. 상대적 날짜 표현("3일 전", "다음 주")을 사용하지 않는다. 절대 날짜(2024-01-15)를 사용한다.
2. 레이블과 함께 날짜 형식에 대한 설명 텍스트를 제공한다 (예: "YYYY-MM-DD").
3. 년, 월, 일 각 입력 필드에 개별 레이블을 제공한다.
4. 날짜 형식에 대한 도움말을 입력 필드 근처에 표시한다.
5. 시작일/종료일 등 여러 날짜 입력이 필요한 경우 원본과 동일한 구성을 사용한다.
6. 대략적인 날짜 입력(월만, 년만)이 가능하도록 허용한다 (해당되는 경우).
7. 적절한 경우 자동완성을 지원한다 (생년월일 등).
8. **상호작용**: 직접 입력 또는 달력 피커를 통한 날짜 선택을 모두 지원한다.

### 접근성 가이드라인

- 각 입력 필드에 `<label>`을 연결한다 ("년", "월", "일").
- 전체 날짜 입력 그룹에 `<fieldset>`과 `<legend>`를 사용한다.
- 달력 피커 버튼에 `aria-label="달력 열기"`를 제공한다.
- 오류 메시지를 `aria-describedby`로 연결한다.
- KWCAG 2.2 / WCAG 2.1 준수: 1.3.1 Info and Relationships, 3.3.2 Labels or Instructions, 3.3.5 Help

### 상호작용

- **Tab**: 년→월→일 필드 순서로 포커스 이동
- **입력**: 숫자만 입력 가능
- **달력 아이콘 클릭**: 달력 피커 열기
- **달력에서 날짜 선택**: 선택한 날짜가 입력 필드에 반영

### 플랫폼 고려사항

- 모바일에서 `inputmode="numeric"` 속성으로 숫자 키보드를 표시한다.
- 네이티브 날짜 피커(`<input type="date">`) 대신 KRDS 달력 컴포넌트를 사용한다.

## 파일 업로드 (File Upload)

### 용도

사용자가 파일을 선택하여 업로드하는 컴포넌트. 단일 또는 다중 파일 업로드를 지원한다.

### HTML 스니펫

```
html/code/file_upload.html
```

### 구조

- 파일 업로드 컨테이너 (`krds-file-upload`)
  - 레이블 (`<label>`)
  - 파일 유형/크기/개수 안내 텍스트
  - 파일 선택 버튼
  - 업로드된 파일 목록
    - 파일명
    - 삭제 버튼
  - 오류 메시지 (선택)

### 사용성 가이드라인

1. 모달 내에서 다중 파일 업로드를 사용하지 않는다. 모달에서는 단일 파일만 허용한다.
2. 업로드된 파일 항목에는 파일명과 삭제 버튼을 함께 표시한다.
3. 파일 업로드 영역에 명확한 레이블을 제공한다.
4. 허용되는 파일 유형, 최대 크기, 최대 개수를 안내 텍스트로 표시한다.
5. 파일 업로드 후에도 업로드 버튼은 기본 상태로 유지한다 (추가 업로드 가능).
6. 파일 선택 버튼은 적절한 강조 수준을 사용한다 (Secondary 이하).
7. 파일 선택 시 자동으로 폼을 제출하지 않는다. 사용자가 별도로 제출 버튼을 클릭해야 한다.
8. 파일 유형/크기 오류 시 명확한 오류 메시지를 표시한다 (예: "PDF 파일만 업로드 가능합니다").
9. 긴 파일명은 한 줄에 말줄임(...)으로 표시한다.
10. **탐색 상호작용**: 파일 선택 버튼에 포커스/호버 시 시각적 피드백을 제공한다.
11. **업로드 상호작용**: 파일 선택 후 목록에 추가, 진행률 표시 (대용량 파일의 경우).
12. **삭제 상호작용**: 삭제 버튼 클릭 시 해당 파일을 목록에서 제거한다.

### 접근성 가이드라인

- 파일 입력(`<input type="file">`)에 `<label>`을 연결한다.
- `accept` 속성으로 허용 파일 유형을 명시한다.
- 삭제 버튼에 `aria-label="파일명 삭제"`를 제공한다.
- 파일 목록 변경 시 스크린 리더에 알린다 (`aria-live="polite"`).
- 오류 메시지를 `aria-describedby`로 연결한다.
- KWCAG 2.2 / WCAG 2.1 준수: 1.3.1 Info and Relationships, 3.3.1 Error Identification, 3.3.2 Labels or Instructions

### 상호작용

- **파일 선택 버튼 클릭/Enter**: 파일 탐색기 열기
- **파일 선택**: 파일 목록에 추가
- **삭제 버튼 클릭/Enter**: 해당 파일 제거
- **Tab**: 파일 선택 버튼, 파일 목록 항목, 삭제 버튼 간 포커스 이동
- **드래그 앤 드롭**: 파일을 영역에 드래그하여 업로드 (선택적)

### 플랫폼 고려사항

- 모바일에서 카메라 촬영 옵션을 제공할 수 있다 (`capture` 속성).
- 모바일에서 파일 목록의 터치 영역을 충분히 확보한다.
